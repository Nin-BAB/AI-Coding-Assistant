## 二、STAR 法则 — 详细版

### Situation（背景）
"在面试前，我发现很多研发同学在编码过程中反复切换 AI 工具和 IDE，效率不高。我想做一个把 AI 能力直接嵌入研发流程的工具。"

### Task（任务）
"我需要完成一个端到端的 AI 辅助编码工具，覆盖从需求到代码再到 Git 提交的全流程。而且不只是一个脚本，要能服务化、要有界面、要能接入实际开发。"

### Action（行动）
1. **第一天**：先用 OpenAI SDK 调用 DeepSeek API，实现了代码生成、代码分析、Bug 修复三个核心功能，CLI 交互验证
2. **第二天**：升级到 LangChain 框架，用 `@tool` 定义工具（Python 执行、文件读取等），用 `create_agent` 构建了可自主调用工具的 Agent；打通 Git 工作流（自动生成 Commit Message + 代码审查）；用 FastAPI 封装成 REST 服务
3. **第三天**：用 Gradio 搭建多 Tab 交互界面，降低使用门槛

### Result（结果）
"完成了一个生产级可用的 AI 编程工具，覆盖研发全流程。整个过程中，我对 LLM 应用开发、Agent 原理、服务化部署都有了深入理解。"

## 三、面试高频 Q&A

### Q1: 为什么用 LangChain 而不用直接调 API？
> "我第一天就是直接调 API 的手动 ReAct 循环，但发现要自己解析 AI 输出（TOOL:/ANSWER:），自己维护循环逻辑，相当麻烦。LangChain 的 `create_agent` 帮我把 Agent 循环、工具调用、状态管理都封装好了，我只需要关注业务逻辑——定义 `@tool` 和写好系统提示词就行。"

**考点**：证明你有「先手动实现，再抽象升级」的过程，不是拿来就用。

### Q2: @tool 装饰器是怎么工作的？
> "`@tool` 是 LangChain 的装饰器，把普通函数变成 Agent 可用的工具。函数名自动成为工具名，docstring 成为工具描述（LLM 根据描述决定何时调用），参数类型注解成为输入 Schema。Agent 在推理时会根据用户请求和已有结果，自主决定调用哪个工具、传什么参数。"

### Q3: System Prompt 和 User Message 的区别？
> "System Prompt 是给模型设定角色和规则的——告诉它你是谁、能干什么、不能干什么。User Message 是用户具体的问题。System Prompt 优先级更高，模型会优先遵循系统指令。我们的编码 Agent 在 System Prompt 里定义了工作流程和规则，然后用 User Message 传入用户的实际问题。"

### Q4: 为什么要拆 CLI / API / UI 三个层？
> "三层设计是渐进式的：CLI 最快验证 AI 能力是否可用；FastAPI 把能力服务化，让其他系统可以调用；Gradio UI 降低使用门槛，非技术人员也能用。面试官问就是『前后端分离思想 + 渐进式开发策略』。"

### Q5: Git Commit 自动生成是怎么实现的？
> "核心流程：先用 `git diff HEAD` 获取代码变更——`subprocess.run` 执行命令行取结果，再把 diff 发给 LLM，用规范化的 System Prompt 要求它按 Conventional Commits 格式生成提交信息。这样每次 commit 前跑一下，提交信息就规范了。"

### Q6: 遇到的最大技术难点？
> "最大的坑是 **Agent 的工具调用可靠性**。早期直接调 API 做 ReAct 时，`exec()` 执行代码字符串如果代码有语法错误就直接崩了。解决办法有两个：1）用 LangChain 的 `create_agent` 替代手动循环，框架自动处理工具调用的结果反馈；2）在每个工具函数内部加 try-except，保证工具出错时有报错信息返回而不是直接死掉。"

## 五、面试官可能会深挖的问题

1. **LangChain create_agent 跟旧版 create_react_agent 有什么区别？**
   - 答：create_agent 基于 LangGraph（有向图），create_react_agent 是旧版循环。新版的图结构更灵活（可加条件分支、人工审核节点）。

2. **exec() 执行用户代码安全吗？**
   - 答：不安全。`exec()` 是 Python 里最危险的操作之一，生产环境要用沙箱（Docker 容器隔离或 Pyodide 浏览器执行）。我们项目里只用于本地调试验证。

3. **FastAPI 为什么比 Flask 更适合？**
   - 答：异步支持（性能更好）、Pydantic 集成（自动数据校验）、自动生成 OpenAPI 文档（`/docs` 开箱即用）。

4. **Gradio 跟 Streamlit 你了解吗？**
   - 答：Gradio 更偏 AI demo 场景，集成度高，ChatInterface 开箱即用；Streamlit 更灵活但代码量大。选 Gradio 因为项目是面向 AI 应用的，开发效率更高。
